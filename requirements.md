# Требования к приложению TaskManager

## 1. Функциональные требования

### 1.1. Добавление задачи
- Приложение должно позволять добавлять задачи с полями:
  - **ID**: автоматически генерируемый уникальный идентификатор
  - **Название**: текстовое поле, обязательное
  - **Описание**: текстовое поле, опциональное
  - **Статус**: по умолчанию "новая"

### 1.2. Просмотр списка задач
- Приложение должно отображать список всех задач
- Должна быть возможность просмотра задач с фильтрацией по статусу

### 1.3. Изменение статуса задачи
- Приложение должно поддерживать изменение статуса задачи
- Статусы: "новая" → "в работе" → "завершена"
- Изменение статуса должно быть доступно через меню

### 1.4. Редактирование задачи
- Приложение должно позволять редактировать название и описание существующей задачи
- Редактирование должно происходить по ID задачи

### 1.5. Удаление задачи
- Приложение должно поддерживать:
  - Одиночное удаление задачи по ID
  - Массовое удаление задач (по нескольким ID или по критериям)

### 1.6. Автосохранение при выходе
- При выходе из приложения все изменения должны автоматически сохраняться в файл tasks.json
- При запуске приложения данные должны автоматически загружаться из tasks.json

## 2. Нефункциональные требования

### 2.1. Хранение данных
- Все данные должны храниться в файле **tasks.json** в корне проекта
- Формат данных: JSON с отступами (indent=4) для читаемости
- Если файл не существует, приложение должно создавать его при первом сохранении

### 2.2. Обработка ошибок
- Приложение не должно падать при неверном вводе пользователя
- При неверном вводе должно выводиться понятное сообщение об ошибке
- Приложение должно корректно обрабатывать отсутствие файла tasks.json (возвращать пустой список)

### 2.3. Производительность
- Приложение должно быстро загружаться
- Операции с задачами должны выполняться без заметных задержек

### 2.4. Удобство использования
- Интерфейс должен быть интуитивно понятным
- Меню должно быть структурированным и понятным
- Сообщения об ошибках должны быть информативными

## 3. User Stories

### US-1: Добавление задачи
**Как** пользователь,  
**Я хочу** добавить задачу "Сдать проект",  
**Чтобы** не забыть дедлайн.

**Критерии приемки:**
- Пользователь может ввести название и описание задачи
- Задача сохраняется с уникальным ID
- Статус задачи устанавливается как "новая"

### US-2: Просмотр статуса задач
**Как** пользователь,  
**Я хочу** видеть статус задач,  
**Чтобы** понимать прогресс выполнения.

**Критерии приемки:**
- Пользователь может просмотреть список всех задач
- Для каждой задачи отображается текущий статус
- Статусы отображаются понятным образом

### US-3: Изменение статуса задачи
**Как** пользователь,  
**Я хочу** изменить статус задачи с "новая" на "в работе",  
**Чтобы** отслеживать прогресс выполнения.

**Критерии приемки:**
- Пользователь может выбрать задачу по ID
- Пользователь может изменить статус на следующий в цепочке
- Изменение статуса сохраняется в файл

### US-4: Редактирование задачи
**Как** пользователь,  
**Я хочу** отредактировать название или описание задачи,  
**Чтобы** уточнить детали или исправить ошибку.

**Критерии приемки:**
- Пользователь может выбрать задачу по ID
- Пользователь может изменить название и/или описание
- Изменения сохраняются в файл

### US-5: Удаление завершенных задач
**Как** пользователь,  
**Я хочу** удалить все завершенные задачи,  
**Чтобы** очистить список от выполненных задач.

**Критерии приемки:**
- Пользователь может выбрать массовое удаление
- Приложение удаляет все задачи со статусом "завершена"
- Изменения сохраняются в файл

### US-6: Сохранение данных
**Как** пользователь,  
**Я хочу** чтобы мои задачи сохранялись автоматически,  
**Чтобы** не потерять данные при закрытии приложения.

**Критерии приемки:**
- Все изменения автоматически сохраняются при выходе
- При следующем запуске все задачи загружаются из файла
- Файл tasks.json создается автоматически при первом сохранении
